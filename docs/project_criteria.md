# Критерии оценивания курсового проекта

## 1. Перечень обязательных сдаваемых артефактов

Студент предоставляет следующие артефакты в составе Git-репозитория:

| № | Артефакт | Описание | Формат |
|---|----------|----------|--------|
| 1 | **Репозиторий** | Исходный код проекта, структурированный и документированный | Git (GitHub/GitLab) |
| 2 | **README.md** | Общее описание проекта: цель, задачи, структура репозитория, быстрый старт | Markdown |
| 3 | **PROJECT_BRIEF.md** | Детальная постановка задачи: целевая аудитория, сценарии использования, метрики успеха | Markdown |
| 4 | **architecture.md** | Архитектурный обзор: компоненты системы, потоки данных, диаграммы (C4, компонентная, UML) | Markdown + диаграммы (PlantUML/Mermaid) |
| 5 | **Инструкция по запуску** | Пошаговое руководство по установке зависимостей, настройке и запуску проекта | README.md или docs/setup.md |
| 6 | **Демо-материалы** | Jupyter Notebook с демонстрацией работы, графики, тестовые сценарии, скриншоты/видео | .ipynb, .png, .mp4 |
| 7 | **Презентация** | Слайды для защиты проекта (до 8 минут доклада) | .pdf или .pptx |
| 8 | **Материалы оценки качества** | Метрики, результаты экспериментов, сравнение методов, выводы | .csv, .json, Markdown-таблицы |

## 2. Критерии и веса оценивания

Итоговая оценка курсового проекта рассчитывается по формуле:

**I = Σ (баллᵢ × весᵢ)**, где Σ весᵢ = 100%.

| Критерий | Ось | Вес | Что оценивается |
|----------|-----|-----|-----------------|
| **Постановка задачи и обоснование выбора методов Монте-Карло** | Программное решение | 15% | Чёткость формулировки задачи; обоснованность выбора методов (MCMC, HMC, NUTS, MCTS, Particle Filters и др.); соответствие предметной области |
| **Работа с данными** | Программное решение | 15% | Качество генерации синтетических данных; предобработка реальных данных; работа с пропусками и выбросами; оценка качества данных |
| **Архитектура и MVP-прототип** | Программное решение | 20% | Полнота реализации MVP; корректность интеграции методов Монте-Карло; устойчивость к ошибкам; обработка особых случаев |
| **Эксперименты и качество** | Программное решение | 10% | Релевантность метрик (сходимость MCMC, R-hat, ESS, эффективный размер выборки); сравнение методов; воспроизводимость |
| **Репозиторная документация** | Документация | 15% | README, PROJECT_BRIEF, архитектурный обзор, инструкции запуска; полнота и качество описания |
| **Воспроизводимость и инженерная культура** | Документация | 10% | Структура репозитория; версионирование зависимостей (requirements.txt, environment.yml); тесты (опционально); CI/CD (опционально) |
| **Анализ ограничений, рисков и практической ценности** | Документация | 10% | Границы применимости; риски (вырождение частиц, сходимость MCMC, выбор proposal distribution); направления развития; ценность для предметной области |
| **Соответствие индикаторам роли (КРМ)** | Сквозной (обе оси) | 5% | Явное соответствие индикаторам выбранной «профильной» ИИ-роли (см. ролевой лист руководителя) |

## 3. 10-балльная рубрика по осям

### 3.1. Ось «Программное решение (MVP)»

| Балл | Требования к уровню |
|------|----------------------|
| **10** | Выдающийся проект: все компоненты MVP реализованы и протестированы; эксперименты содержат сравнительный анализ нескольких методов (например, Metropolis-Hastings vs. HMC vs. NUTS); обоснованы архитектурные и инженерные решения; присутствуют признаки научной новизны или высокой практической ценности. |
| **8** | Проект реализован полностью и качественно: MVP устойчив; методы Монте-Карло интегрированы обоснованно; эксперименты проведены по нескольким метрикам; чёткое соответствие требованиям. |
| **6** | MVP реализован в базовом объёме: методы Монте-Карло работоспособны; есть эксперименты по одной метрике (например, сходимость MCMC); часть требований проработана. |
| **4** | Минимально приемлемый проект: MVP реализует базовый сценарий, но с существенными вопросами к качеству реализации или обоснованности выбора методов. |
| **0–3** | MVP отсутствует или неработоспособен; постановка задачи или выбор методов не обоснованы. |

### 3.2. Ось «Документация»

| Балл | Требования к уровню |
|------|----------------------|
| **10** | Документация уровня публикации: полный набор артефактов (README, PROJECT_BRIEF, architecture.md, ADR, инструкции, демо-сценарии). Проект полностью воспроизводим внешним экспертом. |
| **8** | Полный комплект документации: репозиторий структурирован; README информативен; есть архитектурный обзор и инструкции запуска. Отдельные разделы требуют незначительной доработки. |
| **6** | Документация присутствует в достаточном объёме: README, краткое описание архитектуры, инструкции запуска. Отдельные разделы неполные. |
| **4** | Минимальная документация: есть README и краткое описание, но воспроизведение возможно только с помощью автора. |
| **0–3** | Документация отсутствует или существенно неполна; проект невоспроизводим. |

## 4. Правило нижнего порога

**Условие получения зачёта по курсовому проекту:**
- **Не менее 4 баллов по каждой оси** («Программное решение» и «Документация»).
- Если студент получает < 4 баллов по одной из осей, проект считается **незачтенным** и направляется на доработку.
