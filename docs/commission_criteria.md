# Критерии защиты курсового проекта перед комиссией

## 1. Регламент защиты

Защита проекта проводится перед комиссией в составе 3–5 членов (председатель, преподаватель дисциплины, приглашённые эксперты, представители индустрии). 

**Таймбоксы защиты:**

| Этап | Длительность | Что происходит |
|------|--------------|----------------|
| **Доклад студента** | до 8 минут | Презентация проекта по обязательным элементам (см. раздел 2) |
| **Живая демонстрация MVP** | до 5 минут | Запуск кода, демонстрация работы алгоритмов, визуализация результатов |
| **Вопросы комиссии** | до 10 минут | Ответы на вопросы по проекту, методам, коду, экспериментам |
| **Заполнение оценочных листов и обсуждение** | до 5 минут | Члены комиссии заполняют формы, выставляют баллы |
| **Общее время на одну работу** | до 30 минут | |

**Примечания:**
- При невозможности живой демонстрации допускается предзаписанное видео (не более 3 минут) с сохранением возможности вопросов по коду и данным.
- Использование подготовленных шпаргалок, телефонов и других электронных устройств **запрещено**, кроме демонстрационного ноутбука.
- Студент может использовать бумажные заметки только при ответах на вопросы (по решению комиссии).

---

## 2. Обязательные элементы презентации

Презентация должна содержать **не менее 7 слайдов** (или разделов), соответствующих перечню:

| № | Элемент | Содержание |
|---|---------|------------|
| 1 | **Титульный слайд** | Название проекта, ФИО студента, группа, профильная ИИ-роль, ФИО руководителя |
| 2 | **Постановка задачи** | Проблема, целевая аудитория, сценарии использования, метрики успеха (предметные и ML) |
| 3 | **Обоснование выбора методов Монте-Карло** | Почему выбран именно этот метод (MCMC, HMC, MCTS и др.)? Альтернативы и их ограничения |
| 4 | **Архитектура решения и потоки данных** | Схема компонентов, диаграмма потоков данных |
| 5 | **Демонстрация работы MVP** | Скриншоты, видео, результаты работы кода на тестовых данных (2–3 примера) |
| 6 | **Результаты экспериментов** | Метрики качества, сравнение с бейзлайном, анализ сходимости, графики, таблицы |
| 7 | **Ограничения, риски и дальнейшее развитие** | Границы применимости метода, возможные улучшения, этические аспекты (если применимо) |
| 8 | **Соответствие профильной ИИ-роли КРМ** | Как проект демонстрирует индикаторы заявленной роли (ссылка на КРМ) и планы развития компетенций |
| 9 | **Заключительный слайд** | Выводы, благодарности, ссылки на репозиторий и контакты |

**Примечание:** Слайды с демонстрацией и экспериментами могут быть объединены, но общее количество разделов должно быть не менее 7.

## 3. Оценочный лист члена комиссии

Каждый член комиссии заполняет оценочный лист по 8 критериям (таблица ниже). Итоговый балл защиты (D) вычисляется как взвешенная сумма:

**D = Σ (баллᵢ × весᵢ)**, где Σ весᵢ = 100%.

| Критерий | Вес | Балл (0–10) | Комментарий |
|----------|-----|-------------|-------------|
| **C1: Постановка задачи и актуальность** | 10% | | Насколько чётко сформулирована проблема, определён целевой пользователь и метрики успеха? |
| **C2: Обоснование выбора методов Монте-Карло** | 10% | | Почему выбран именно этот метод? Рассмотрены ли альтернативы и их ограничения? |
| **C3: Архитектура решения и работа с данными** | 15% | | Логичность архитектуры, качество работы с данными (генерация/предобработка), интеграция методов |
| **C4: Демонстрация MVP и качество эксперимента** | 20% | | Работоспособность кода, полнота демонстрации, релевантность метрик, сравнительный анализ |
| **C5: Качество презентации** | 10% | | Структура, визуальное оформление, наглядность графиков, соблюдение тайминга |
| **C6: Владение материалом, ответы на вопросы** | 15% | | Глубина понимания методов, способность объяснить математические основы, аргументированные ответы |
| **C7: Инженерная культура и воспроизводимость** | 10% | | Качество кода, документация, воспроизводимость результатов, использование Git |
| **C8: Соответствие профильной ИИ-роли КРМ** | 10% | | Насколько проект подтверждает индикаторы заявленной роли (см. ролевой блок) |

**Итоговый балл защиты** = среднее арифметическое баллов всех членов комиссии (или медиана — по решению председателя), округлённое до целого.

## 4. Расшифровка уровней 0–10 для защиты

| Балл | Характеристика уровня защиты (для всех критериев) |
|------|-----------------------------------------------------|
| **10** | Выдающаяся защита: чёткая логика, глубокое владение материалом, живая демонстрация без замечаний, ответы уровня специалиста-практика, признаки научной новизны или высокой практической ценности. |
| **9** | Отличная защита: развёрнутые аргументированные ответы, качественная демонстрация, комиссия имеет только уточняющие вопросы. |
| **8** | Уверенная защита: полные и корректные ответы, качественные слайды, демонстрация проходит с минимальными замечаниями. |
| **7** | Хорошая защита: аргументированные ответы на большинство вопросов, отдельные пробелы в деталях реализации. |
| **6** | Достаточная защита: основные ответы даны, ответы на глубокие вопросы неполные, демонстрация проходит по подготовленному сценарию. |
| **5** | Слабая защита: студент справляется только с ответами по подготовленному тексту, демонстрация ограничена. |
| **4** | Минимально приемлемая защита: демонстрирует базовое понимание, испытывает затруднения с вопросами о данных, коде и метриках. |
| **0–3** | Не защищена: не может воспроизвести собственный проект, не отвечает на существенные вопросы комиссии, демонстрация отсутствует или неработоспособна. |

## 5. Ролевой блок защиты (по КРМ)

Критерий C8 предполагает соотнесение представленной работы с «профильной» ИИ-ролью студента. Комиссия определяет фактический уровень по 10-балльной шкале, опираясь на индикаторы и «сигналы» защиты.

### 5.1. ML Engineer (Инженер машинного обучения)

| Уровень | Сигналы при защите |
|---------|---------------------|
| **10** | Демонстрация продвинутой инженерной архитектуры, использование современных библиотек (PyMC, PyTorch) с оптимизацией производительности, обоснование выбора hyperparameters. |
| **8** | Качественная реализация MVP, показан код, обсуждаются метрики и выбор proposal distribution, интеграция с внешними данными. |
| **6** | Код работает, но есть проблемы с архитектурой (неоптимальная структура, отсутствие обработки ошибок), метрики выбраны формально. |
| **4** | Код запускается только по инструкции, студент не может объяснить внутреннее устройство алгоритмов, метрики не интерпретированы. |

### 5.2. Data Analyst (Аналитик данных)

| Уровень | Сигналы при защите |
|---------|---------------------|
| **10** | Глубокий DA, визуализация распределений, проверка статистических гипотез, обоснование выбора данных, интерпретация неопределённости (bayesian credible intervals). |
| **8** | Проведён качественный анализ данных, есть визуализации сходимости MCMC, интерпретация результатов в предметной области. |
| **6** | Базовый DA выполнен, но визуализации неполные, выводы поверхностны. |
| **4** | Анализ данных отсутствует или выполнен формально, студент не может объяснить статистические показатели. |

### 5.3. ML Researcher (Исследователь МО)

| Уровень | Сигналы при защите |
|---------|---------------------|
| **10** | Сравнение нескольких методов с теоретическим обоснованием, анализ сходимости, предложение собственных модификаций алгоритмов, ссылки на литературу. |
| **8** | Обоснован выбор метода, проведён анализ сходимости, сравнение с бейзлайном, ссылки на статьи. |
| **6** | Метод выбран интуитивно, анализ сходимости формален (только графики без количественных метрик). |
| **4** | Студент не понимает теоретических основ алгоритма, не может объяснить условие детального баланса или сходимость. |
